Git 笔记

git 配置

Git 提供了一个叫做 git config 的工具，专门用来配置或读取相应的工作环境变量。

这些环境变量，决定了 Git 在各个环节的具体工作方式和行为。这些变量可以存放在以下三个不同的地方：

  • /etc/gitconfig 文件：系统中对所有用户都普遍适用的配置。若使用 git config 时用 --system 选项，读写的就是这个文件。
  • ~/.gitconfig 文件：用户目录下的配置文件只适用于该用户。若使用 git config 时用 --global 选项，读写的就是这个文件。
  • 当前项目的 Git 目录中的配置文件（也就是工作目录中的 .git/config 文件）：这里的配置仅仅针对当前项目有效。每一个级别的配置都会覆盖上层的相同配置，所以 .git/config 里的配置会覆盖 /etc/gitconfig 中的同名变量。

在 Windows 系统上，Git 会找寻用户主目录下的 .gitconfig 文件。主目录即 $HOME 变量指定的目录，一般都是 C:\Documents and Settings$USER。

此外，Git 还会尝试找寻 /etc/gitconfig 文件，只不过看当初 Git 装在什么目录，就以此作为根目录来定位。

关于git区域

在 Git 中，有三个主要区域用于管理代码的状态，它们是工作目录（Working Directory）、暂存区（Staging Area），以及版本库（Commit）。

  1. Working Directory（工作目录）：

    • 含义： 工作目录是你当前正在工作的项目文件夹，包含了你的项目中的所有文件和子目录。
    • 状态： 在工作目录中，你可以对文件进行编辑、创建新文件、删除文件等操作。这些更改未被 Git 跟踪，直到你将它们添加到暂存区。

  2. Staging Area（暂存区）：

    • 含义： 暂存区是一个中间区域，用于暂时存放你希望在下一次提交时包含的更改。
    • 状态： 当你使用 git add 命令将文件添加到暂存区时，Git 将记录这些更改，并在下一次提交时将它们包含在版本库中。

  3. Commit（版本库）：

    • 含义： 版本库是存储项目所有历史记录的地方，包括每一次提交的快照。
    • 状态： 当你使用 git commit 命令提交暂存区的更改时，Git 会创建一个新的提交，将暂存区中的更改保存为项目历史的一部分。

git rm --cached <file> 和 git restore --staged <file> 都是用于从暂存区中移除文件，但它们有不同的目的和效果。

  1. git rm --cached <file>：

    • 目的： 用于将文件从工作目录和暂存区中移除，同时保留在工作目录中。
    • 效果： 文件会从暂存区中移除，但保留在工作目录中，意味着工作目录中的文件不会被删除。你需要手动删除工作目录中的文件，如果希望从版本控制中移除这个文件。

  2. git restore --staged <file>：

    • 目的： 用于将文件从暂存区中撤销，保留在工作目录中。
    • 效果： 撤销暂存区中对文件的更改，但保留在工作目录中。这使得文件不再处于暂存状态，但仍存在于工作目录，你可以继续在工作目录中对文件进行修改。

总结：

  • 如果你想要从版本控制中完全移除某个文件，包括工作目录中的文件，你应该使用 git rm --cached <file>，并在适当的时候手动删除工作目录中的文件。

  • 如果你只是想暂时取消对文件的跟踪并将其从暂存区中撤销，而保留在工作目录中，你可以使用 git restore --staged <file>。

分支的常用命令总结

  • git branch: 列出所有分支。
  • git branch <branch-name>: 创建新分支。
  • git checkout <branch-name> 或 git switch <branch-name>: 切换分支。
  • git switch -c <branch-name>: 创建并切换到新分支。
  • git merge <branch-name>: 合并指定分支到当前分支。
  • git branch -d <branch-name>: 删除已合并的分支。
  • git branch -D <branch-name>: 强制删除未合并的分支。

这是一个简单的 Git 分支教程，涵盖了常见的分支操作。Git 分支是一个非常强大的工具，能够有效地组织和管理项目的开发流程。
